# QA 로그·보고서 자동화 기준

> 최초 작성일: 2026-07-17
> 상태: **공통 누적 로그·최신 실행 요약·manifest·도메인별 정식 보고서 분리 구현 및 비API 검증 완료**

## 목적

품질검사 관리 GUI의 모든 시험이 같은 방식으로 실행 증적을 남기도록 한다. 원문 로그는 실행할 때마다 새 파일로 누적하고, 공통 실행 요약은 가장 최근 실행 결과로 갱신한다. 공통 실행 요약과 도메인별 정식 결과 보고서는 역할을 분리한다.

## 적용 범위

- AI 상담 품질검사 8개 시험
- 고객 의견 분석 품질검사 전체 비AI 검사와 단위 테스트
- 고객 의견 분석 A~D 에이전트 교차 테스트
- 완료·실패·사용자 중지·시작 실패 실행

## 저장 규칙

```text
_OUTPUT/logs/qa/qa_runs.jsonl
_OUTPUT/logs/qa/runs/{test_id}/{run_id}.log
_OUTPUT/logs/qa/k6/{test_id}/{run_id}_summary.json

_OUTPUT/reports/qa/latest/{test_id}.md
_OUTPUT/reports/qa/latest/latest_report.md
_OUTPUT/reports/manifests/qa/{test_id}.json
_OUTPUT/reports/manifests/qa/latest.json
```

- `run_id`는 실행 시각과 임의 식별자를 결합해 중복을 방지한다.
- `qa_runs.jsonl`은 시작·종료 이벤트를 계속 추가한다.
- 실행별 `.log`는 명령, 설정, 전체 콘솔 출력, 종료 상태를 보존하며 덮어쓰지 않는다.
- k6 원본 요약도 실행별 파일로 누적한다.
- 시험별 Markdown은 정식 품질 보고서가 아니라 최근 실행 상태를 보여주는 보조 요약이며 같은 시험의 최근 결과로 덮어쓴다.
- `latest_report.md`는 시험 종류와 관계없이 가장 마지막에 끝난 QA 실행 요약으로 덮어쓴다.
- manifest는 시험별 최신본과 전체 최신본만 유지한다. 대시보드는 폴더나 파일명을 추측하지 않고 manifest의 `sources`와 `report`를 읽는다.
- VOC A~D 실행 요약은 해당 프로필의 정식 `quality_score_report.md`를 함께 연결한다.

## 공통 기록 항목

- 실행 식별자, 시험 ID와 시험명
- 실행 명령과 실행 조건
- 시작·완료 시각과 총 실행시간
- 완료·실패·사용자 중지·시작 실패 상태
- 프로세스 종료 코드와 오류 내용
- 원문 로그, k6 요약, 최신 보고서 연결 정보

부하 시험은 최대 가상 인원(VU)과 실행 시간을 기록한다. A~D 에이전트 교차 테스트는 프로필, 실행 시점의 전체 테스트케이스 수와 실제 AI 평가 대상 수를 기록한다.

서버 연결 성능 종합 시험은 `1명 → 10명 → 25명`의 실행 단계, 단계별 독립 실행 방식, 단계 사이 5초 안정화 대기, 정상 완료 기준 총 36건의 실제 채팅 요청을 기록한다. 성능 원본 데이터의 각 요청에는 단계 식별자를 저장하고, 최신 성능 보고서는 시작 시각 구간이 아니라 이 식별자를 기준으로 단계별 결과를 집계한다.

VOC 테스트케이스 정식 보고서의 단일 실행·로그·표·그래프 기준은 `VOC_TESTCASE_REPORT_AUTOMATION.md`를 따른다.

AI Agent 실제 챗봇 대화의 백그라운드 채점과 매 채팅 최신 실시간 보고서 자동 갱신은 QA GUI 공통 실행 요약과 별개이며 `AI_AGENT_LIVE_REPORT_AUTOMATION.md`를 따른다.

## 결과 지표

k6 요약이 존재하면 다음 값을 보고서와 manifest에 기록한다.

- 전체 HTTP 요청 수
- 요청 실패율
- 평균 응답시간
- p95 응답시간
- 검사 성공·실패 수

pytest 출력이 존재하면 통과·실패·오류·제외·선택 제외 건수를 기록한다. 별도 수치가 없는 시험도 실행 상태, 실행시간, 종료 코드와 원문 로그를 반드시 남긴다.

## 실패 및 중지 처리

실패하거나 사용자가 중지한 시험도 보고서를 생성한다. k6 요약이 생성되지 않았거나 출력이 일부만 존재해도 확보된 원문 로그와 종료 상태를 보존한다. 최신 보고서 쓰기는 임시 파일을 만든 뒤 교체하는 방식으로 처리해 작성 도중의 불완전한 파일 노출을 줄인다.

프로세스 종료 코드가 0이어도 `Judge 실패`, 외부 AI API 오류, `미평가(API 실패)`가 출력되면 보고서 상태를 `경고 포함 완료`로 기록한다. A~D 프로필 실행기는 각 사례의 Judge CSV를 확인하고 미평가 결과가 있으면 실패 종료 코드를 반환한다.

## 실제 외부 AI 실행 범위

사용자가 QA GUI 또는 통합 대시보드에서 A~D 실행 버튼을 직접 누르면 실행 시점에 등록된 테스트케이스 전체를 사용한다. 확인창에는 전체 케이스 수, 실제 AI 평가 대상 수, 프로필과 예상 호출 범위를 표시한다.

Codex가 개발·통합 검증 명령을 실행할 때는 사용자가 단순 실행을 승인해도 대표 사례 2개만 사용한다. 사용자가 Codex에게 `전체 테스트케이스 실행`을 명시해야만 전체 실행으로 전환한다. A~D 비교 검증도 별도 전체 실행 지시가 없으면 각 프로필에 동일한 대표 2개를 재사용한다.

## 2026-07-17 실제 A 프로필 검증

- 범위: A 프로필, `TC-01`, `TC-02` 두 사례만 사용
- 생성 설정: OpenAI `gpt-5.6-luna`, 추론 없음
- 독립 평가 설정: Anthropic `claude-sonnet-5`, 추론 낮음
- 두 사례의 6개 에이전트 파이프라인과 OpenAI 생성 단계는 완료됐다.
- Anthropic 인증 오류 `401`로 두 사례의 독립 평가는 `미평가(API 실패)`가 됐다.
- 공통 최신 보고서는 해당 실행을 `경고 포함 완료`로 표시하고 누적 원문 로그를 연결한다.
- 인증 오류는 키 값을 문서나 로그에 복사하지 않고 오류 유형만 기록했다.
- Retriever가 호스트의 Windows 기본 CSV 경로를 Docker 내부에서 찾지 못하던 문제는 에이전트 로컬 기본 `voc.csv`로 안전하게 보정하도록 수정했다.
- `llm_judge` 모듈 직접 실행 시 누락됐던 `sys` import를 보완했다.
- 이후 OpenAI·Anthropic 인증 조회는 모두 성공했다. 다만 구조 변경 과정에서는 실제 두 사례를 다시 실행하지 않았다.

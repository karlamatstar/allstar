# 챗봇 서버 상태 감지와 재접속 기준

> 최종 갱신: 2026-07-18
> 상태: **구현·비API 자동검사·실제 독립 중단/복구 검증 완료**

## 1. 목적

AI 에이전트 챗봇과 VOC 챗봇은 서로 다른 서버를 사용한다. 사용자가 각 챗봇 화면에 들어왔을 때 해당 서버가 중단되어 있다면 세션에 남은 상태와 관계없이 실제 Health 결과를 기준으로 동일한 형태의 중단 안내와 재접속 기능을 제공한다.

## 2. 서버 구분

| 화면 | Docker 서비스 | Health 주소 | 영향 범위 |
|---|---|---|---|
| AI 에이전트 챗봇 | `portfolio-api` | `http://localhost:8000/health` | AI 에이전트 챗봇만 잠금 |
| VOC 챗봇 | `voc-api` | `http://localhost:8100/health` | VOC 챗봇만 잠금 |

한 서비스만 중단되면 다른 챗봇은 계속 사용할 수 있다. Docker 자체나 Streamlit이 중단된 경우에는 이 화면에서 복구할 수 없다.

## 3. 공통 화면 동작

- 각 챗봇 화면 진입 시 즉시 Health를 확인한다.
- 화면을 열어 둔 동안에는 3초 간격으로 해당 서버 상태를 확인한다.
- 서버 중단 시 기존 대화 기록은 그대로 유지한다.
- 채팅창 중앙에 서버 중단 제목, 답변 불가 안내, 재접속 버튼을 표시한다.
- 채팅 입력창과 전송 버튼을 잠근다.
- AI 화면은 503·504·서버 중단 시험 버튼 3개도 잠근다.
- VOC 화면은 A~D 프로필 선택 버튼도 잠근다.
- 재접속 성공은 Health HTTP 200으로 판정하고 중앙 성공 안내를 표시한다.
- 재접속 뒤 일반 채팅 입력은 기존 외부 API 비용 확인 체크 상태를 다시 따른다.

## 4. 화면별 차이

### AI 에이전트 챗봇

- 명시적인 `채팅 서버 중단 시험`을 유지한다.
- 시험으로 발생한 장애는 실제 대화·채점 로그와 보고서에 `N/A`로 기록한다.
- 재접속 수명주기는 장애 이벤트 로그에도 기록한다.

### VOC 챗봇

- 인위적으로 서버를 중단하는 시험 버튼은 제공하지 않는다.
- 실제 `voc-api` 중단만 감지한다.
- `VOC 채팅 서버 재접속`은 `docker compose up -d voc-api`를 실행한다.
- 일반 운영 복구이므로 AI 장애 시험 로그에는 기록하지 않는다.
- 처리 중이던 요청은 `voc-api` 재시작 뒤 메모리 상태가 보존되지 않으므로 재접속 성공 시 화면의 진행 요청을 정리한다.

## 5. 제한사항

현재 재접속 버튼은 호스트에서 실행 중인 Streamlit이 Docker 명령을 사용할 수 있다는 전제를 따른다. 나중에 Streamlit을 Docker 안에서 실행하면 호스트 Docker 제어 기능을 별도로 연결하거나 전용 서버 관리 API를 두어야 한다.

## 6. 2026-07-18 검증 결과

- 관련 자동검사 11개와 대시보드 모듈 컴파일이 통과했다.
- `voc-api`만 중단했을 때 AI Health 200을 유지하면서 VOC 화면만 중앙 경고·입력 잠금·A~D 버튼 4개 잠금으로 전환됐다.
- VOC 화면의 재접속 버튼으로 `voc-api` Health 200과 중앙 완료 안내를 확인했다.
- `portfolio-api`만 외부에서 중단했을 때 VOC Health 200을 유지하면서 AI 화면만 중앙 경고·입력 및 장애 버튼 3개 잠금으로 전환됐다.
- AI 화면의 재접속 버튼으로 `portfolio-api` Health 200과 장애 버튼 3개 재활성화를 확인했다.
- 실제 외부 AI API와 K6 시험은 실행하지 않았다.

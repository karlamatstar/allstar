# 컨트롤러 중복 실행 차단 기준

> 작성일: 2026-07-17
> 대상: AllStar 서버 관리, 품질검사 관리 GUI
> 상태: 구현·자동 검증 완료

## 1. 목적

서버 관리 또는 품질검사 관리 프로그램이 이미 실행 중일 때 같은 프로그램의 두 번째 창을 만들지 않는다. 두 번째 실행은 경고창만 표시하고 사용자가 `확인`을 누르면 새로 시작한 프로세스만 종료한다.

## 2. 사용자 동작

1. 서버 관리 또는 품질검사 관리 프로그램을 정상 실행한다.
2. 같은 프로그램의 `.bat`, 숨김 `.vbs` 또는 Python 진입점을 다시 실행한다.
3. `이미 실행 중입니다` 제목의 경고창을 표시한다.
4. 서버 관리는 `서버 관리 프로그램이 이미 실행 중입니다.`, 품질검사 관리는 `품질검사 관리 프로그램이 이미 실행 중입니다.`라고 안내한다.
5. 사용자가 `확인`을 누르면 두 번째 프로세스만 종료하며 기존 창과 실행 중인 서버·시험은 유지한다.

서버 관리와 품질검사 관리는 서로 다른 프로그램이므로 각각 한 개씩 동시에 실행할 수 있다.

## 3. 구현 방식

- `src/allstar/shared/single_instance.py`의 Windows 이름 있는 뮤텍스를 공통 사용한다.
- 서버 관리는 `Local\\AllStarServerControlCenter`, 품질검사 관리는 `Local\\AllStarQAControlCenter`라는 서로 다른 이름을 사용한다.
- 잠금은 GUI 프로세스가 살아 있는 동안 유지되며 정상 종료와 예외 종료 시 핸들을 닫는다.
- 프로세스가 강제로 종료되면 Windows가 뮤텍스 핸들을 정리하므로 다음 정상 실행을 막는 오래된 잠금 파일이 남지 않는다.
- 중복 프로세스는 전용 종료 코드 `23`을 반환한다.

## 4. 서버 종료 감시기 예외 처리

서버 관리 런처는 `shutdown_guard.py`가 GUI 종료를 감시한다. 두 번째 서버 관리 실행이 중복 차단으로 끝난 경우 정상 종료 표식이 없더라도 강제 종료로 판단하면 안 된다.

- 종료 코드가 `23`이면 `중복 실행 차단` 로그만 남긴다.
- `stop_project_services`를 호출하지 않는다.
- 첫 번째 서버 관리 창과 기존 Streamlit·Docker 서비스는 그대로 유지한다.
- 실제 비정상 종료인 경우에만 기존 강제 종료 정리 흐름을 수행한다.

## 5. 검증

- 공통 잠금의 최초 획득과 해제를 가짜 Windows API로 검증한다.
- 이미 존재하는 뮤텍스 핸들을 즉시 닫고 중복 상태를 반환하는지 검증한다.
- 실제 별도 Python 프로세스 두 개에서 같은 Windows 뮤텍스를 두 번째 프로세스가 획득하지 못하는지 검증한다.
- 서버·품질검사 진입점의 경고창 문구와 서로 다른 잠금 이름을 확인한다.
- 서버 종료 감시기의 중복 종료 분기에서 서비스 종료 함수를 호출하지 않는지 확인한다.

중복 실행 잠금 자체의 검증에는 실제 서버 시작·외부 AI API 호출이 필요하지 않다. 후속 Streamlit 종료 오류 수정 검증에서는 로컬 Streamlit만 잠깐 실행해 상위·자식 프로세스와 8501 포트가 모두 해제되는 것을 확인했다.

2026-07-17 검증 결과 집중검사 `43개 통과`, 외부 AI·실제 서버 종단 파일을 제외한 전체 비API 회귀 `211개 통과`였다.

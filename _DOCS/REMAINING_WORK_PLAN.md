# AllStar 통합 프로젝트 남은 작업 계획

> 작성일: 2026-07-17
> 기준 프로젝트: `D:\_Study_Project\_Total`

## 현재 상태

디렉터리 구조 개편, Docker·Streamlit 서버 관리 GUI, QA 관리 GUI, A~D 모델 프로필, 기본 통합 대시보드와 비AI 회귀 검증은 완료했다. 남은 작업은 시험 결과를 표준화하고 대시보드·모니터링에서 확인할 수 있게 만드는 단계다.

구현·검증·문서화·실제 API·Git 반영을 분리한 전체 진행 상태는 `PROJECT_PROGRESS_CHECKLIST.md`에서 관리한다.

## 1. QA 실행 안전성 보완

상태: **구현 완료, 비AI 검증 완료**

- 한 번에 하나의 QA 시험만 실행하도록 전체 탭 공통 잠금을 사용한다.
- 시험 실행 중에는 모든 실행 버튼을 비활성화하고 현재 시험의 중지 버튼만 활성화한다.
- 중지 명령은 별도 작업에서 처리해 GUI가 멈추지 않게 한다.
- 시작·완료·실패·사용자 중지·시작 실패 상태를 `_OUTPUT/logs/services/launcher/qa_control_runs.jsonl`에 JSONL로 기록한다.
- 장애·기능 검증 시험의 기본 pytest에서는 실제 외부 AI 호출 파일과 `end_to_end` 시험을 제외한다.
- 실제 AI 시험은 QA 안전 검증에 섞지 않고 7번 절차에서 별도 승인 후 실행한다.

완료 기준:

- AI Agent QA와 VOC QA 전체 14개 시험 탭에 공통 잠금 적용
- 실행 중 다른 탭에서 새 시험을 시작할 수 없음
- 실행 상태가 화면과 JSONL 로그에 일치하게 기록됨
- 장애·기능 검증 기본 실행에 실제 OpenAI·Anthropic 호출이 포함되지 않음

## 2. QA 보고서 자동 생성 통일

상태: **구현 완료, 비AI 자동 검증 완료**

대상:

- 기본 동작 시험
- 일반 부하 시험
- 무작위 요청 시험
- 한계 부하 시험
- 순간 급증 시험
- 서버 연결 끊김 방어 시험

구현 내용:

- 시험 종류, 실행 시각, VU, 실행시간, 성공·실패·중지 상태를 공통 형식으로 저장
- 요청 수, 실패율, 평균·p95 응답시간 등 k6 결과 표준화
- 원본 결과와 요약 보고서를 연결하는 실행 식별자 및 manifest 생성
- 실패하거나 중지된 실행도 부분 결과를 보존
- 대시보드가 폴더를 직접 추측하지 않고 공통 manifest를 읽도록 기준 확정
- VOC A~D GUI 실행은 등록된 전체 사례를 프로필별 정식 보고서 하나로 통합
- 실행 JSON 로그에서 Markdown·CSV·JSON과 PNG 그래프를 재생성
- 최신 프로필 결과가 2개 이상이면 A~D 종합 비교 보고서 자동 갱신

상세 저장 규칙과 공통 필드는 `QA_REPORT_AUTOMATION.md`를 따른다. 실행별 원문 로그와 k6 요약은 계속 누적하고, 시험별 보고서·전체 최신 보고서·manifest는 최근 실행 결과로 덮어쓴다.

전체 탭 배치, 기능 보존, GUI 전체 케이스 실행과 Codex 대표 2건 검증 분리 기준은 `INTEGRATED_DASHBOARD_STRUCTURE_REQUIREMENTS.md`를 따른다.

- 신규 VOC 챗봇 보고서와 VOC A~D 테스트케이스 보고서 2개 탭 추가

- 보고서 본문을 파일 경로 노출 없이 화면에서 확인
- AI Agent와 VOC 결과 분리

6개 보고서 탭의 데이터 원본, 화면 규칙과 완료 기준은 `INTEGRATED_REPORT_DASHBOARD_REQUIREMENTS.md`를 따른다.

## 5. 챗봇 대시보드 세부 개선

상태: **AI 챗봇 5개 하위 탭과 VOC A~D 테스트케이스별 7단계 실시간 상태·완료 결과 구현**

남은 작업:

- 사용자 승인 후 A~D 중 선택 프로필의 전체 실제 실행 화면을 장시간 관찰
- 실제 실행에서 실패 단계·재시도·검색 결과 0건 안내가 의도대로 전환되는지 확인
- 단계별 처리시간을 별도 비교 표로 확장할지는 실제 화면 확인 후 결정

구현 방식은 별도 API 이벤트 스트림 대신 실행별 `run_id` 진행 JSON을 사용한다. 호스트의 Interpreter·LLM Judge와 Docker의 Summarizer가 같은 `_OUTPUT` 경로를 공유하고, 대시보드는 VOC 실행 영역만 주기적으로 부분 갱신한다.

남은 작업:

## 7. 실제 AI 대표 검증

상태: **A 프로필 대표 2건 실행 완료, Anthropic 인증 오류로 독립 평가 재검증 필요**

실행 원칙:

- 첫 검증은 A 프로필과 대표 사례 `TC-01`, `TC-02`만 사용한다.
- 실행 전에 선택 사례, 모델 프로필, 예상 외부 호출 횟수를 다시 확인한다.
- A 결과 확인 후 필요할 때만 B~D에 동일한 두 사례를 재사용한다.
- 이 제한은 Codex가 개발·검증 명령을 실행할 때 적용한다. 사용자가 GUI·대시보드에서 직접 누르는 실행은 등록된 전체 테스트케이스를 사용한다.
- Codex의 전체 사례 또는 A~D 대량 실행은 사용자가 `전체 테스트케이스 실행`을 명시하지 않으면 수행하지 않는다.

2026-07-17에 승인된 A 프로필 `TC-01`, `TC-02`를 실행했다. 두 사례의 파이프라인과 OpenAI 생성은 완료됐으나 당시 Anthropic 인증 오류로 Judge가 미평가됐다. 이후 OpenAI·Anthropic 인증 조회는 모두 성공했다. 변경된 통합 보고서 구조의 실제 재검증은 같은 두 사례와 대상 프로필을 다시 확인한 뒤 진행하며, B~D는 별도 승인 전까지 실행하지 않는다.

## 권장 진행 순서

```text
1. QA 실행 안전성 완료
2. QA 보고서 자동 생성 통일
3. 통합 대시보드 전체 상위 구조
4. AI·VOC 챗봇 및 테스트케이스 화면
5. 모니터링 Grafana 4개와 리포트 6개 하위 탭
6. 대시보드 반응형·빈 상태 검증
7. 실제 AI 대표 검증
```

AWS 또는 외부 공개 배포는 위 작업과 별도로 인증, HTTPS, QA 권한, 부하 제한, 감사 로그 정책을 다시 검토한 뒤 진행한다.

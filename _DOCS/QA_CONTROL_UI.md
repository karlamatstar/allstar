# 품질검사 관리 GUI 화면 기준

> 최초 작성일: 2026-07-17
> 상태: **GUI·실행 잠금·중지·K6 안내·단일 실행 차단 구현 및 비API 검증 완료**

> 후속 구현사항: AI 상담 품질검사 마지막에 `테스트케이스 품질 시험 (Test Case Test)`을 추가했고 직접 부하 K6 5개는 Grafana 중심으로 운영한다. 장애·기능 검증 및 서버 연결 성능 정식 보고서는 유지한다. 상세 기준은 `QA_AI_TESTCASE_TAB_AND_K6_REPORT_POLICY.md`를 따른다.

## 프로그램 중복 실행 차단

- 품질검사 관리가 이미 실행 중이면 두 번째 GUI 창을 만들지 않는다.
- 두 번째 실행에는 `이미 실행 중입니다` 경고창과 `확인` 버튼을 표시한다.
- `확인`을 누르면 새로 실행한 프로세스만 종료하며 기존 시험과 기존 창은 유지한다.
- 서버 관리와는 별도 잠금을 사용하므로 서버 관리 한 개와 품질검사 관리 한 개는 동시에 실행할 수 있다.

공통 잠금과 검증 기준은 `CONTROL_CENTER_SINGLE_INSTANCE.md`를 따른다.

## 탭 표시 원칙

- 탭은 쉬운 한국어 기능명을 첫째 줄에 표시한다.
- 영문 전문용어가 필요하면 괄호와 함께 둘째 줄에 표시한다.
- 선택 탭과 비선택 탭은 `14×9`의 동일한 내부 여백을 사용한다.
- 선택 상태는 탭 크기를 바꾸지 않고 배경색과 글자색으로만 구분한다.
- 탭의 글자는 가로·세로 중앙에 정렬한다.

최상위 탭은 다음처럼 표시한다.

```text
AI 상담 품질검사
(AI Agent QA)

고객 의견 분석 품질검사
(VOC QA)
```

AI 상담 품질검사의 영문 시험명도 `기본 동작 시험`과 `(Smoke Test)`처럼 두 줄로 나눈다. 장애 모의 k6, 전체 기능 검사 pytest, 결함 보고서 생성을 함께 수행하는 탭은 `장애·기능 검증 시험`과 `(Validation Test)`로 표시한다.

## 고객 의견 분석 A~D 탭

A~D는 단순히 `A 테스트`로 표시하지 않고 다음 형식을 사용한다.

```text
에이전트 교차 테스트
(A)
```

B, C, D도 같은 형식으로 표시한다. 탭 안쪽의 화면 제목은 `에이전트 교차 테스트 (A)`처럼 한 줄로 표시하고, 답변 생성 모델·독립 품질 평가 모델·추론 강도 설명은 기존대로 유지한다.

## AI 테스트케이스 품질 시험 탭

- AI 상담 품질검사의 기존 탭 뒤 가장 마지막에 `테스트케이스 품질 시험`과 `(Test Case Test)`를 두 줄로 표시한다.
- 기존 시험과 동일하게 `실행`, `중지`, 아래 터미널 화면을 사용한다.
- QA 컨트롤러에서 실행하면 등록된 AI 테스트케이스 전체를 사용하고 외부 API 비용 안내를 먼저 확인한다.
- 정상 완료 시 기존 AI 테스트케이스 정식 보고서와 그래프를 갱신한다.
- 중지·실패 시 실행 로그는 남기되 마지막 정상 정식 보고서는 덮어쓰지 않는다.

## 실행 버튼과 결과 위치

- 각 탭에는 `실행`, `중지` 버튼만 둔다.
- `보고서 폴더` 버튼은 제공하지 않는다.
- 실행 결과는 아래 실행 기록 화면에 표시한다.
- 생성된 보고서는 `_OUTPUT/reports/` 구조에 저장한다.

## 공통 실행 잠금과 상태 기록

- AI 상담 품질검사 8개와 고객 의견 분석 품질검사 6개, 총 14개 시험 탭은 하나의 공통 실행 잠금을 사용한다.
- 시험 하나가 실행 중이면 모든 탭의 실행 버튼을 비활성화한다.
- 현재 실행 중인 탭의 중지 버튼만 활성화한다.
- 시험이 끝나면 모든 실행 버튼을 다시 활성화하고 중지 버튼을 비활성화한다.
- 시작·완료·실패·사용자 중지·시작 실패 상태는 `qa_control_runs.jsonl`에 구조화해 기록한다.
- 장애·기능 검증 시험은 기본적으로 실제 외부 AI 호출 테스트를 제외하고 비AI 기능 회귀만 수행한다.
- 모든 시험의 전체 콘솔 출력은 실행별 원문 로그로 누적한다.
- 직접 부하 K6 5종을 제외한 시험은 종료 상태와 확보된 결과로 공통 최신 Markdown 보고서와 manifest를 자동 갱신한다.
- k6 시험은 요청 수·실패율·평균·p95 응답시간을 자동 기록한다.
- 종료 코드가 0이어도 Judge·외부 AI 실패가 포함되면 최신 보고서에 `경고 포함 완료`로 표시한다.
- 상세 규칙은 `QA_REPORT_AUTOMATION.md`를 따른다.
- 공통 QA Markdown은 실행 요약으로 사용하고, VOC A~D GUI 실행은 등록된 전체 테스트케이스가 합쳐진 프로필별 정식 보고서를 별도로 생성해 실행 요약에서 연결한다.
- VOC A~D 실행 중 사용자 중지·서버 종료·API/파이프라인 실패가 발생하면 실행별 로그와 초안만 보존하고 마지막 정상 프로필 정식 보고서·그래프·manifest는 유지한다.
- VOC 정식 보고서의 로그·표·그래프 규칙은 `VOC_TESTCASE_REPORT_AUTOMATION.md`를 따른다.

GUI 화면에서는 보고서의 파일 경로나 폴더 위치를 표시하지 않는다. 이후 대시보드에서 보고서를 확인하는 흐름을 기준으로 하며, 각 탭의 상단 설명 영역은 저장 위치 대신 다음 내용을 안내한다.

- 이 시험이 어떤 방식으로 요청을 발생시키는지
- 이 시험으로 어떤 상태와 품질을 확인할 수 있는지
- 시험 중 어떤 단계가 진행되는지
- 결과 또는 보고서가 자동으로 정리·생성되는지

기본 동작·일반 부하·무작위 요청·한계 부하·순간 급증·장애 및 기능 검증·서버 연결 성능 시험은 각각 별도의 설명을 사용한다. VOC 전체 비AI 검사·단위 테스트·A~D 에이전트 교차 테스트에도 해당 실행 범위에 맞는 설명을 표시한다.

2026-07-17 후속 정리에서 `서버 연결 끊김 방어 시험`과 전용 실행 스크립트를 제거했다. AI 에이전트 챗봇 질문의 `퇴근`·`강사` 단어로 각각 503·504 장애를 강제로 만들던 규칙과 공개 요청 필드 `simulate_api_disconnect`도 함께 제거했다. 실제 통신 장애가 자연적으로 발생했을 때의 재시도와 안전한 대체 응답 처리는 유지한다.

Streamlit 통합 대시보드는 실제 API 요청 전에 화면 안의 `필수 체크 사항` 확인란을 사용한다. QA 컨트롤러는 시험별 실행 범위·요청 수·부하 또는 외부 API 호출 내용을 실행 직전 별도 팝업으로 다시 확인하는 기존 방식을 유지한다. 두 화면 모두 사용자가 확인하기 전에는 실제 시험을 시작하지 않는다.

K6를 사용하는 기본 동작·일반 부하·무작위 요청·한계 부하·순간 급증·장애 및 기능 검증·서버 연결 성능 시험은 화면 설명에서 `K6 부하 시험 도구`를 명시한다. K6가 없으면 일반적인 실행 파일 오류를 보여주지 않고, K6 다운로드·설치가 필요하다는 한국어 안내창을 먼저 표시한다. 사용자가 동의하면 웹 브라우저로 Grafana K6 공식 설치 페이지를 연다.

직접 부하 시험인 기본 동작·일반 부하·무작위 요청·한계 부하·순간 급증 5개는 터미널·원본 로그·K6 JSON·Prometheus·Grafana 결과만 유지하고 별도 사용자용 보고서를 만들지 않는다. K6와 Pytest를 결합한 장애·기능 검증의 결함 보고서와 1명·10명·25명 서버 연결 성능 보고서는 기존대로 유지한다.

## K6 실행 파일 기준

- K6는 Python 패키지가 아니라 별도의 외부 실행 프로그램이므로 저장소의 소스 코드만 받아서는 자동으로 생기지 않는다.
- 실행 시 프로젝트의 `RUN/k6.exe`를 먼저 찾고, 없으면 Windows 시스템 설치 경로(`PATH`)의 `k6`를 찾는다.
- 공식 설치 안내 페이지는 `https://grafana.com/docs/k6/latest/set-up/install-k6/`이다. Windows에서는 공식 문서가 안내하는 `winget install k6 --source winget` 명령이나 최신 공식 설치 파일을 사용할 수 있다.
- 기존 프로젝트도 같은 탐색 구조였으며 실제 `k6.exe`는 Git에 포함되어 있지 않았다. `_Total` 통합 시 K6 스크립트와 호출 코드는 이전됐지만 별도 설치 단계는 수행되지 않았다.
- `k6.exe`는 `.gitignore` 때문에 제외되거나 최근 작업에서 삭제된 것이 아니다. 현재 저장소의 Git 이력에도 포함된 적이 없다.

## 서버 연결 성능 종합 시험

- 가상 사용자 `1명 → 10명 → 25명` 순서로 실제 `/chat` 요청을 실행한다.
- 한 번의 k6 실행에 세 단계를 예약하지 않고, 각 인원 단계를 별도의 k6 프로세스로 독립 실행한다.
- 앞 단계의 모든 요청이 종료된 뒤 서버 안정화를 위해 5초 대기하고 다음 단계를 시작한다.
- 응답이 무기한 멈추는 상황을 방지하기 위해 각 단계의 최대 실행시간은 2분으로 제한한다. 제한에 도달하면 해당 단계를 종료한 뒤 안정화 대기를 거쳐 다음 단계로 진행한다.
- 각 요청에는 `phase1`, `phase2`, `phase3` 단계 정보를 직접 기록한다. 보고서가 시작 시각 기준으로 단계를 추측하지 않으므로 응답이 느려져도 단계별 결과가 섞이지 않는다.
- 정상 완료 시 요청 수는 1건, 10건, 25건으로 총 36건이다. 이 시험은 모의 응답이 아닌 실제 AI 상담 `/chat` API를 호출하므로 실행 전 확인이 필요하다.
- 최신 성능 보고서에는 단계별 완료 요청 수와 평균 응답시간, 전체 p95 응답시간, 오류율을 기록한다.

## 부하 시험 입력값

이전 `ai_agent_quality_portfolio/RUN/test_launcher.py`의 부하 설정을 유지한다.

| 시험 | 기본 최대 가상 인원(VU) | 기본 실행 시간 |
|---|---:|---:|
| 일반 부하 시험(Load Test) | 20명 | 60초 |
| 무작위 요청 시험(Random Test) | 100명 | 60초 |
| 한계 부하 시험(Stress Test) | 100명 | 120초 |
| 순간 급증 시험(Spike Test) | 200명 | 60초 |

- 네 시험 탭에는 `최대 가상 인원(VU)`과 `실행 시간(초)` 입력란을 표시한다.
- 두 값은 모두 1 이상의 정수만 허용한다.
- 실행 전 확인 창과 실행 기록에 실제 적용값을 표시한다.
- 실행 프로세스에는 `K6_VUS`, `SCRIPT_DURATION`, `TARGET_IP=127.0.0.1:8000` 환경변수로 전달한다.
- 모든 K6 실행에는 Prometheus 실시간 전송 출력과 실행별 `testid`를 자동으로 추가한다.
- `K6_PROMETHEUS_RW_SERVER_URL` 기본값은 로컬 Prometheus의 `http://127.0.0.1:9090/api/v1/write`이며, p95 패널을 위해 추세 통계에 `p(95)`를 포함한다.
- 기본 동작 시험(Smoke Test)은 VU 1 고정이므로 입력란을 표시하지 않는다.

## 2026-07-17 검증

- Tkinter GUI를 숨김 상태로 생성해 최상위·AI 시험·VOC 시험 탭의 실제 두 줄 문자열을 확인했다.
- 선택·비선택 탭의 실제 여백 값이 모두 `(14, 9)`인지 확인했다.
- QA 화면 전용 테스트를 포함한 통합 테스트 결과는 구현 기준서의 최신 검증 기록을 따른다.
- 공통 실행 잠금과 버튼 상태 전환, 구조화 JSONL 기록, 실제 AI 기본 제외 조건을 확인했다.
- 통합 테스트 42개가 통과했다.
- 단계별 독립 성능 시험과 K6 설치 안내 검증을 포함해 실제 외부 AI 호출 테스트를 제외한 비AI 회귀 테스트 149개가 통과하고 2개가 선택 제외됐다.
- 승인된 A 프로필 `TC-01`, `TC-02`를 실제 실행했다. 파이프라인 생성은 완료됐지만 Anthropic 인증 오류로 독립 평가는 미평가됐으며 최신 보고서에 `경고 포함 완료`로 기록했다.
- 현재 개발 컴퓨터에는 Windows Package Manager로 Grafana K6 `2.1.0`을 설치했다. 실행 파일은 `C:\Program Files\k6\k6.exe`에 있으며 QA 컨트롤러의 시스템 경로 탐색으로 인식되는 것을 확인했다.
- K6 설치는 현재 컴퓨터의 환경 상태이며 Git 저장소에 실행 파일을 포함하지 않는다. 다른 컴퓨터에서는 K6를 별도로 설치해야 한다.
- K6 2.1.0 기본 동작 시험 1회에서 출력 대상이 Prometheus remote write로 표시되고 `k6_http_reqs_total` 2건이 수집되는 것을 확인했다.
- Grafana·K6 후속 개선 직후의 당시 비API 회귀는 `233개 통과·1개 환경 제외·2개 선택 제외`였다.
- 인위적 장애 유발 기능 제거 후 집중검사 `39개 통과`, 최신 비API 회귀는 `250개 통과·1개 환경 제외·2개 선택 제외`다. AI 서버 컨테이너의 `/health` 200과 공개 요청 스키마에서 강제 장애 필드가 사라진 것도 확인했다.

# VOC 챗봇 품질 화면·100점 채점·검색 실패 개선 계획

> 작성일: 2026-07-17
> 상태: **구현·비API 회귀·브라우저·실제 A~D API 검증 완료**
> 대상: 통합 대시보드 `VOC 챗봇`, VOC 실시간 API, 실시간 로그·보고서

## 1. 목적

VOC 챗봇의 실시간 대화 화면에도 대화 로그와 품질 분석 화면을 추가하고, VOC 테스트케이스에서 사용하는 9항목·100점 루브릭을 동일하게 적용한다. 2026-07-17 실제 질문 처리 중 발생한 검색 0건 실패와 잘못된 단계 표시도 함께 개선 대상으로 관리한다.

이 문서는 최초 문제 분석과 2026-07-17 구현·검증 결과를 함께 보존하는 기준 문서다.

## 2. 확정된 VOC 챗봇 하위 탭

VOC 챗봇에는 다음 5개 하위 탭을 둔다.

1. `챗봇과 대화`
2. `대화 로그`
3. `품질 현황`
4. `유형별 비교`
5. `대화별 채점 상세`

### 2.1 대화 로그

- 가장 최근 대화를 위에 표시한다.
- 로컬 시간, 질문, 사용 프로필 A~D, 생성 모델, 평가 모델, 응답, 처리시간, 총점, 판정과 처리 상태를 보여준다.
- 요청 ID는 기본 표에서 숨기고 기술 상세에서만 조회할 수 있게 한다.
- 실패·데이터 없음·Judge 미실행은 품질 실패로 바꾸지 않고 `N/A`로 구분한다.

### 2.2 품질 현황

- 전체 대화 수와 정상 채점 수를 분리한다.
- 100점 기준 평균, 판정별 건수, 평균 및 필요 시 p95 처리시간을 표시한다.
- 같은 질문을 A~D로 각각 실행했다면 프로필별 결과를 좌우 비교하기 쉽게 한 묶음으로 표시한다.
- 기본 최근 10개이며 5개·10개·20개·전체 선택, 이전·다음 이동을 제공한다.

### 2.3 유형별 비교

- A~D는 답변 생성 모델 하나가 아니라 `생성 모델 + 독립 평가 모델` 조합이므로 화면에 `A~D 프로필 조합 비교`라고 명시한다.
- 왼쪽에는 9개 항목 레이더 그래프, 오른쪽에는 정확한 평균 점수와 평가 건수 표를 배치한다.
- 항목별 최대 점수가 15점·10점·5점으로 다르므로 레이더 그래프는 각 항목을 0~100%로 정규화한다.
- 정확한 점수표에는 `13/15`, `8/10`, `4/5`처럼 원점수와 최대 점수를 함께 표시한다.
- 비교 대상 프로필의 데이터가 없으면 0점으로 그리지 않고 `데이터 없음`으로 표시한다.

### 2.4 대화별 채점 상세

- 가장 최근 채점 시각이 맨 위에 오도록 정렬한다.
- 질문, 답변, A~D 프로필, 생성·평가 모델, 총점, 판정, 처리시간을 표시한다.
- 9개 항목의 점수·최대 점수·항목별 근거와 종합 근거를 표시한다.
- AI 에이전트의 `대화별 채점 상세`, 테스트케이스의 `케이스 상세`와 화면 구조는 맞추되 VOC의 9항목·100점 루브릭을 사용한다.

## 3. VOC 공통 9항목·100점 채점 기준

실시간 VOC 챗봇과 VOC 테스트케이스는 다음 하나의 공통 루브릭을 사용한다.

| 항목 | 최대 점수 | 확인 내용 |
|---|---:|---|
| Interpreter 해석 정확성 | 15 | 질문 의도와 검색 조건이 적절한가 |
| Retriever 검색 관련성 | 15 | 관련 VOC를 제대로 검색했는가 |
| Summarizer 사실성·요약성 | 15 | 왜곡·누락 없이 핵심을 정리했는가 |
| Evaluator 평가 타당성 | 10 | 평가 근거가 일관적인가 |
| Critic 위험 탐지력 | 10 | 실제 문제와 한계를 지적했는가 |
| Improver 실행 가능성 | 15 | 정책 개선안이 구체적이고 실행 가능한가 |
| Agent 연계 품질 | 10 | 앞 Agent 결과를 다음 Agent가 적절히 활용했는가 |
| 장애 대응·로그 | 5 | 오류를 숨기지 않고 추적 가능한 정보를 제공하는가 |
| 성능 | 5 | 전체 응답시간이 수업·발표 시연에 허용 가능한가 |
| **합계** | **100** | |

판정 기준은 공통 `judge_rubric.json`을 따른다.

- 90점 이상: 배포 가능
- 80–89점: 조건부 배포 가능, 개선 후 재검증
- 70–79점: 주요 개선 필요
- 69점 이하: 배포 보류
- 즉시 보류 조건 발생: 점수와 별개로 배포 보류(즉시)
- API·Judge 실패 또는 평가 자료 부족: `N/A`

## 4. 개선 전 구현에서 발견한 채점 기준 불일치

개선 전 VOC 테스트케이스는 `src/allstar/voc/evaluation/judge_rubric.json`의 9항목·100점 기준을 사용한 반면, 실시간 VOC 챗봇의 `src/allstar/voc/api/judge.py`는 `relevance`, `groundedness`, `usefulness`, `safety` 네 항목을 각 1~5점으로 채점했다.

Git 기록상 이 4항목 코드는 2026-07-16 초기 통합 구현 커밋 `d175369a`에서 처음 들어왔다. 해당 기준을 결정한 요구사항이나 문서 기록은 없으며 사용자가 승인한 설계가 아니다. 따라서 별도 VOC 기준으로 유지하지 않고 제거해야 한다.

현재 구현에는 다음 원칙을 적용했다.

- 실시간 전용 4항목 프롬프트를 폐기한다.
- 실시간과 테스트케이스가 공통 루브릭, 프롬프트 생성, 응답 검증과 판정 함수를 공유한다.
- 화면에서만 100점으로 환산하지 않고 실제 Judge 출력·로그·보고서 스키마를 9항목으로 변경한다.
- 이전 4항목 로그는 과거 스키마로 구분하고 새 100점 데이터와 단순 평균하지 않는다.

## 5. 2026-07-17 실제 VOC 챗봇 처리 실패

### 5.1 실행 조건

| 구분 | 값 |
|---|---|
| 사용자 질문 | 보험 가입 방법에 대한 문제점을 정리해봐주세요 |
| 실행 시각 | 2026-07-17 17:01:24~17:01:29 KST |
| 요청 ID | `fefdbae15e334b71b0c6bb3e38282078` |
| 프로필 | A · 기본 권장 |
| 생성 모델 | OpenAI `gpt-5.6-luna`, 추론 `none` |
| 평가 모델 | Anthropic `claude-sonnet-5`, 추론 `low` |
| 외부 호출 결과 | 생성 파이프라인 실패, 독립 Judge 미실행 |
| 저장 상태 | 대화 로그 `failed`, Judge `N/A` |

### 5.2 확인된 실행 흐름

1. VOC API는 `/chat` 요청을 정상 접수하고 작업 ID를 반환했다.
2. Interpreter 호출은 2.50초 후 정상 완료됐다.
3. Retriever는 0.01초 동안 검색했으나 결과가 0건이었다.
4. Summarizer 내부 파이프라인은 검색 결과 없음으로 `ok=False`를 반환했다.
5. Summarizer·Evaluator·Critic·Improver와 독립 Judge는 실행되지 않았다.
6. 전체 요청은 5.676초 후 `failed`로 로그와 최신 VOC 챗봇 보고서에 저장됐다.

저장된 trace는 다음 의미를 갖는다.

```text
Timing:Interpreter=2.50s
Timing:Retriever=0.01s
Retriever:count=0
Timing:AgentPipeline=0.02s
Timing:TotalPipeline=2.54s
```

### 5.3 확정 원인과 유력 원인

확정된 직접 원인은 **Retriever 검색 결과 0건**이다. Interpreter 서비스 장애나 생성 모델 시간 초과가 아니다.

데이터 원본 `src/allstar/voc/data/voc.csv`에는 보험 가입·보험 청약 관련 VOC가 실제로 존재한다. 예를 들어 모바일 보험 가입 입력 반복, 온라인 보험 청약 중 오류, 가입 채널별 가격 차이 사례가 있으므로 질문 주제에 해당하는 데이터 자체가 전혀 없는 상황은 아니다.

현재 실시간 `run_with_question`은 Interpreter가 만든 필터를 그대로 사용하며 `extra_filters=None`으로 실행된다. 원본 질문에서 보조 검색어를 만드는 안전망은 코드에 존재하지만 QA 실행처럼 호출자가 명시한 경우에만 적용되고 실시간 챗봇 기본 경로에는 적용되지 않는다. 따라서 **Interpreter가 지나치게 구체적이거나 데이터 문구와 맞지 않는 필터를 만들었고, 0건일 때 원문 기반 재검색이 없어 그대로 종료됐을 가능성**이 높다.

다만 실패 로그에는 Interpreter가 실제로 만든 `intent_json`과 필터가 저장되지 않았다. 정확한 필터가 없으므로 위 내용은 코드와 데이터에 근거한 유력 원인이며 확정 원인으로 단정하지 않는다.

### 5.4 함께 발견한 관찰 가능성·화면 문제

- 파이프라인이 `ok=False`이면 VOC API가 결과를 작업 상태에 저장하기 전에 예외로 바꾸므로 `intent_json`, 실제 필터와 단계 결과가 유실된다.
- 대시보드는 작업 전체가 `failed`이면 무조건 1단계 Interpreter를 `실패`, 나머지를 `건너뜀`으로 표시한다.
- 실제로는 Interpreter가 완료되고 Retriever에서 0건이 발생했으므로 첨부 화면의 1단계 실패 표시는 사실과 다르다.
- 사용자 답변 영역에 기술 trace 전체가 `처리 실패` 문구로 노출되어 일반 사용자가 원인을 이해하기 어렵다.
- 자동 보고서는 실패와 Judge `N/A`를 남기지만, 결과 객체가 없어서 검색 필터·단계별 상태를 보여주지 못한다.

## 6. 실패 처리 개선 기준

### 6.1 검색 0건 재시도

1. Interpreter 필터로 1차 검색한다.
2. 0건이면 원본 질문에서 추출한 안전망 검색어를 병합해 한 번만 제한적으로 재검색한다.
3. 2차 검색에서도 0건이면 시스템 오류가 아니라 `관련 데이터 없음` 상태로 종료한다.
4. 무관한 결과를 억지로 반환하지 않도록 기존 희소어·관련성 기준을 유지한다.

### 6.2 상태와 사용자 안내

- Interpreter 완료는 `완료`로 유지한다.
- Retriever는 호출 오류와 정상적인 0건을 구분해 `실패` 또는 `관련 데이터 없음`으로 표시한다.
- 이후 단계는 정확한 사유를 붙여 `건너뜀`으로 표시한다.
- 사용자에게는 `현재 등록된 VOC 데이터에서 관련 내용을 찾지 못했습니다`와 재질문 안내를 보여준다.
- 기술 trace와 내부 필터는 `실행 내용 보기`에서만 확인하게 한다.

### 6.3 로그와 보고서

- 성공 여부와 관계없이 Interpreter intent, 실제 검색 필터, 재검색 여부, 검색 건수, 단계별 상태·시간을 저장한다.
- 검색 0건은 Judge를 호출하지 않고 점수와 판정을 `N/A`로 기록한다.
- 운영 실패율과 데이터 없음 비율을 서로 다른 지표로 집계한다.
- 실시간 보고서에는 사용자 안내, 기술 원인, 재검색 여부를 구분해서 기록한다.

## 7. 구현 결과

1. 실시간 VOC Judge를 공통 9항목·100점 루브릭으로 전환했다.
2. 기존 4항목·20점 로그는 이전 스키마로 표시하고 새 100점 통계에서 제외했다.
3. 실패와 데이터 없음에서도 intent·필터·재검색·단계 상태를 보존한다.
4. 검색 0건이면 원문 안전망 검색어로 한 번만 재검색한다.
5. `failed`와 `no_data`를 분리하고 Judge 미실행은 `N/A`로 보존한다.
6. 7단계 화면이 실제 진행·실패·데이터 없음 단계와 일치한다.
7. VOC 챗봇에 5개 품질 탭, 최신순, 페이지 이동, A~D 질문 묶음, 레이더와 정확한 점수표를 구현했다.
8. 질문이 `?? ??`처럼 손상되면 외부 API 호출 전에 HTTP 422로 거부한다. 과거 손상 로그는 삭제하지 않고 보고서의 무효 입력 기록에 남기되 품질 통계에서는 제외한다.

## 8. 완료 기준

- 실시간 VOC 챗봇과 테스트케이스가 같은 9항목·100점 루브릭을 사용한다.
- 4항목·20점 실시간 채점이 새 로그에 더 이상 생성되지 않는다.
- 성공 대화는 9개 항목 점수와 100점 총점이 5개 하위 탭과 보고서에 일관되게 표시된다.
- 검색 0건에서도 실제 Interpreter 필터와 재검색 결과를 로그에서 확인할 수 있다.
- 7단계 화면이 실제 실패 단계와 일치한다.
- 데이터 없음은 시스템 실패 또는 품질 FAIL로 집계되지 않는다.
- 실제 API 검증은 대표 질문 2건과 예상 호출 범위를 먼저 확인한 뒤 진행한다.

사용자 승인에 따라 같은 질문 `보험 가입 방법에 대한 문제점을 정리해봐 주세요`를 A~D에 각각 1회 사용했다. 테스트 질문을 추가하지 않았다.

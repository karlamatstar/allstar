> 최초 작성일: 2026-07-17
> 구현 상태: 프로필별 단일 보고서·로그 재생성·표·그래프·종합 비교·정상 완료 보고서 보호와 현재 케이스 확인·수정·변경 전 보관 구현 완료, 비AI 검증 완료

## 목적

VOC A~D 테스트케이스 시험은 프로필 한 번 실행마다 완성된 정식 보고서 한 개를 만든다. 테스트케이스별로 보고서를 나누지 않으며, 실행 로그를 단일 원본으로 사용해 Markdown·CSV·JSON과 그래프를 다시 생성할 수 있어야 한다. 이 결과는 후속 Word 최종 보고서와 발표용 PPT를 만드는 근거 자료로 사용한다.

## 테스트케이스 확인·수정 구현 결과

> 2026-07-17 확정 요구사항을 통합 대시보드에 구현했다.

- 통합 대시보드에서 현재 실행용 테스트케이스를 선택해 전체 정의를 확인하고 수정할 수 있게 한다.
- ID는 기본적으로 유지하며 질문·분류·기대 의도·키워드·필수 출력·금지 출력·평가 설정·예외 설정·참고 사항을 변경할 수 있게 한다.
- 수정 직전 `test_cases.json` 전체를 `archive/revisions/` 아래 날짜·시간 파일로 자동 저장한다.
- 축소 전 20건 원본 `archive/test_cases_20_2026-07-17.json`은 편집 대상에 포함하지 않는다.
- 수정 완료 뒤 대시보드 목록과 전체·AI 평가·장애 전용 건수를 즉시 다시 계산한다.
- 기존 실행 로그와 보고서는 당시 결과로 보존하고 수정된 정의는 다음 실행부터 적용한다.
- 새 실행 로그에는 실행 당시 테스트케이스 스냅샷을 남겨 이후 정의가 바뀌어도 과거 결과를 재현할 수 있게 한다.

변경 저장과 삭제 직전의 현재 실행본 전체는 `src/allstar/voc/evaluation/archive/revisions/test_cases_before_change_YYYYMMDD_HHMMSSffffff.json`으로 보관한다. 새 케이스 추가는 기존 정의를 없애지 않으므로 변경 전 보관을 만들지 않는다. A~D 실행 중에는 추가·수정·삭제 버튼을 모두 잠근다.

## 실행 단위

- 사용자가 QA GUI 또는 통합 대시보드에서 프로필을 실행하면 등록된 VOC 테스트케이스 전체를 사용한다.
- 프로필 하나를 선택하면 `llm_judge` 프로세스 한 번이 전체 사례를 파일 정의 순서대로 처리한다.
- Codex의 개발·검증 실행은 `--case-id`를 사용해 대표 사례 2개로 제한하며, 사용자가 Codex에게 전체 실행을 명시한 경우에만 전체를 사용한다.
- 사례 하나가 끝날 때마다 실행별 로그 폴더의 `report_draft/` 초안을 현재까지의 결과로 갱신한다.
- 전체 대상 처리와 채점 검증이 정상 완료되면 동일 실행 로그로 최종 결과를 재생성한 뒤 프로필 최신 정식 보고서로 한 번에 승격한다.
- 중간 실패·사용자 중지·서버 종료·API 실패 시 완료된 사례와 확보된 데이터는 실행 로그·초안·실행 manifest에 남기고 기존 최신 정식 보고서·그래프·프로필 manifest는 변경하지 않는다.
- `PASS(예외처리)`는 설계된 데이터 없음 정상 결과이므로 게시를 막지 않지만 `미평가(API 실패)`·`미평가(파이프라인 실패)`·파싱 실패는 최종 게시를 막는다.

_OUTPUT/reports/voc/testcase/
├─ a/
│  ├─ quality_score_report.md

_OUTPUT/reports/voc/cross_validation/

├─ 교차검증_그래프데이터.json

- 실행 로그는 `run_id` 폴더를 새로 만들어 계속 누적한다.
- 프로필별 정식 보고서와 데이터·그래프는 해당 프로필의 최근 **정상 완료 실행**으로만 덮어쓴다.
- 테스트케이스별 정식 보고서 폴더를 새로 만들지 않는다.
- 이전 방식으로 생성된 사례별 폴더는 과거 실행 증적으로 유지하지만 새 실행에서 사용하지 않는다.

## 로그 원본 범위

Judge 실행 로그에는 다음 전체 데이터를 보존한다.

- 실행 ID와 시작·완료 시각
- 프로필과 실제 생성·평가 제공자
- 실제 모델과 추론 강도
- 질문과 테스트케이스 ID

- Judge 판정과 상세 근거

- 파이프라인·Judge·전체 처리시간
- API 시도·실패·미평가 사유

최종 정식 보고서는 실행 직후 이 JSON 로그를 다시 읽어 실행별 초안으로 생성하고, 전체 정상 완료를 검증한 뒤 최신 정식 위치로 승격한다. 따라서 Markdown 파일이 손상되거나 Word·PPT를 새로 만들어야 할 때도 로그만으로 표와 그래프를 복구할 수 있다.

## 그래프 생성

그래프는 AI 이미지 생성이 아니라 로그의 수치로 프로그램이 만드는 PNG 파일이다.

- `case_score_comparison.png`: 이번 실행 테스트케이스별 품질 점수
- `criteria_score_rate.png`: 평가 항목별 평균 달성률
- `case_duration_comparison.png`: 파이프라인·Judge·전체 처리시간
- `profile_average_score.png`: A~D 평균 품질 점수
- `profile_average_duration.png`: A~D 평균 처리시간

표·CSV·JSON·그래프는 모두 같은 실행 로그의 데이터를 사용한다. Markdown, Word, PPT는 동일한 PNG를 재사용해 수치와 시각 자료가 어긋나지 않게 한다.

## A~D 실행 결과

A, B, C, D를 각각 한 번씩 실행하면 프로필별 정식 보고서 4개와 종합 비교 보고서 1개가 존재한다.

```text
A 실행 → A 최신 보고서 생성
B 실행 → B 최신 보고서 + A/B 종합 비교 갱신
C 실행 → C 최신 보고서 + A/B/C 종합 비교 갱신
D 실행 → D 최신 보고서 + A/B/C/D 종합 비교 갱신
```

종합 비교는 최신 정식 결과가 두 프로필 이상 있을 때만 생성한다. 프로필 하나만 있으면 비교 보고서를 새로 만들지 않는다. 프로필을 정상 완료하면 해당 프로필 보고서를 승격한 뒤 A~D 최신 정상 결과로 종합 비교를 다시 계산한다. 실패·중지·부분 완료 실행은 종합 비교를 갱신하지 않는다.

## 정상 완료 보고서 보호

2026-07-17 D 재실행 중 Streamlit과 Docker 전체 종료가 겹쳐 TC-05~TC-09가 `6개 에이전트 서버 미가동`으로 실패했다. 당시 Judge가 공식 최신 폴더를 사례마다 직접 갱신해 부분 실패 보고서가 정상 보고서를 덮어썼고, 상위 실행기가 종료되어 manifest와 종합 비교는 이전 정상 실행을 가리키는 불일치가 발생했다.

후속 구현에서는 Judge 출력 위치를 `{run_id}/report_draft/`로 분리했다. 상위 실행기는 프로세스 종료 코드가 0이고 모든 Judge 대상 사례에 API·파이프라인·파싱 실패가 없을 때만 후보 디렉터리를 프로필 최신 정식 디렉터리로 교체한다. 교체 전 실행별 `run_manifest.json`을 기록하며 실패 실행의 초안은 원인 분석용으로 유지한다.

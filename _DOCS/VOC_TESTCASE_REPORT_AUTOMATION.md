# VOC 테스트케이스 정식 보고서 자동화 기준

> 최초 작성일: 2026-07-17
> 구현 상태: 프로필별 단일 보고서·로그 재생성·표·그래프·종합 비교 구현 완료, 비AI 검증 완료

## 목적

VOC A~D 테스트케이스 시험은 프로필 한 번 실행마다 완성된 정식 보고서 한 개를 만든다. 테스트케이스별로 보고서를 나누지 않으며, 실행 로그를 단일 원본으로 사용해 Markdown·CSV·JSON과 그래프를 다시 생성할 수 있어야 한다. 이 결과는 후속 Word 최종 보고서와 발표용 PPT를 만드는 근거 자료로 사용한다.

## 실행 단위

- 사용자가 QA GUI 또는 통합 대시보드에서 프로필을 실행하면 등록된 VOC 테스트케이스 전체를 사용한다.
- 프로필 하나를 선택하면 `llm_judge` 프로세스 한 번이 전체 사례를 파일 정의 순서대로 처리한다.
- Codex의 개발·검증 실행은 `--case-id`를 사용해 대표 사례 2개로 제한하며, 사용자가 Codex에게 전체 실행을 명시한 경우에만 전체를 사용한다.
- 사례 하나가 끝날 때마다 같은 임시 정식 보고서를 현재까지의 결과로 갱신한다.
- 전체 대상 처리가 끝나면 동일 실행 로그를 다시 읽어 최종 정식 보고서를 재생성한다.
- 중간 실패·사용자 중지·API 실패도 완료된 사례와 확보된 데이터를 보고서에 남긴다.

## 저장 구조

```text
_OUTPUT/logs/voc/testcase/
├─ a/{run_id}/llm_judge_{run_id}.json
├─ b/{run_id}/llm_judge_{run_id}.json
├─ c/{run_id}/llm_judge_{run_id}.json
└─ d/{run_id}/llm_judge_{run_id}.json

_OUTPUT/reports/voc/testcase/
├─ a/
│  ├─ quality_score_report.md
│  ├─ llm_judge_result.csv
│  ├─ llm_judge_result.json
│  ├─ report_manifest.json
│  └─ assets/*.png
├─ b/ ...
├─ c/ ...
└─ d/ ...

_OUTPUT/reports/voc/cross_validation/
├─ 교차검증_종합비교보고서.md
├─ 교차검증_그래프데이터.csv
├─ 교차검증_그래프데이터.json
├─ report_manifest.json
└─ assets/*.png
```

- 실행 로그는 `run_id` 폴더를 새로 만들어 계속 누적한다.
- 프로필별 정식 보고서와 데이터·그래프는 해당 프로필의 최근 실행으로 덮어쓴다.
- 테스트케이스별 정식 보고서 폴더를 새로 만들지 않는다.
- 이전 방식으로 생성된 사례별 폴더는 과거 실행 증적으로 유지하지만 새 실행에서 사용하지 않는다.

## 로그 원본 범위

Judge 실행 로그에는 다음 전체 데이터를 보존한다.

- 실행 ID와 시작·완료 시각
- 프로필과 실제 생성·평가 제공자
- 실제 모델과 추론 강도
- 질문과 테스트케이스 ID
- 전체 평가 항목 점수와 총점
- Judge 판정과 상세 근거
- 실제 파이프라인 출력 원문
- 파이프라인·Judge·전체 처리시간
- API 시도·실패·미평가 사유

최종 정식 보고서는 실행 직후 이 JSON 로그를 다시 읽어 생성한다. 따라서 Markdown 파일이 손상되거나 Word·PPT를 새로 만들어야 할 때도 로그만으로 표와 그래프를 복구할 수 있다.

## 프로필별 정식 보고서 구성

1. 보고서 개요와 실행 식별자
2. A~D 프로필 정의표
3. 이번 실행의 실제 모델·추론 설정
4. 이번 실행 전체 테스트케이스 종합 점수표
5. 품질 점수·평가 항목·처리시간 그래프
6. 사례별 질문·실제 답변·항목별 점수·Judge 근거
7. API 실패·미평가·부분 완료 내역
8. 평균 점수와 최종 배포 판정

정상 점수가 없는 `N/A`는 0점으로 바꾸지 않으며 평균과 점수 그래프에서 제외한다. 보고서 표에는 미평가 사유를 그대로 표시한다.

## 그래프 생성

그래프는 AI 이미지 생성이 아니라 로그의 수치로 프로그램이 만드는 PNG 파일이다.

- `case_score_comparison.png`: 이번 실행 테스트케이스별 품질 점수
- `criteria_score_rate.png`: 평가 항목별 평균 달성률
- `case_duration_comparison.png`: 파이프라인·Judge·전체 처리시간
- `profile_average_score.png`: A~D 평균 품질 점수
- `profile_average_duration.png`: A~D 평균 처리시간

표·CSV·JSON·그래프는 모두 같은 실행 로그의 데이터를 사용한다. Markdown, Word, PPT는 동일한 PNG를 재사용해 수치와 시각 자료가 어긋나지 않게 한다.

## A~D 실행 결과

A, B, C, D를 각각 한 번씩 실행하면 프로필별 정식 보고서 4개와 종합 비교 보고서 1개가 존재한다.

```text
A 실행 → A 최신 보고서 생성
B 실행 → B 최신 보고서 + A/B 종합 비교 갱신
C 실행 → C 최신 보고서 + A/B/C 종합 비교 갱신
D 실행 → D 최신 보고서 + A/B/C/D 종합 비교 갱신
```

종합 비교는 최신 정식 결과가 두 프로필 이상 있을 때만 생성한다. 프로필 하나만 있으면 비교 보고서를 새로 만들지 않는다. 프로필을 다시 실행하면 해당 프로필 보고서만 덮어쓰고 종합 비교는 A~D 최신 결과로 다시 계산한다.

## 공통 QA 실행 요약과의 관계

`_OUTPUT/reports/qa/latest/`의 Markdown은 정식 품질 보고서가 아니다. 최근 실행 상태, 실행시간, 오류, 원문 로그와 정식 보고서 위치를 연결하는 보조 요약이다. Word·PPT의 주 원본은 다음 자료다.

- 프로필별 `quality_score_report.md`
- 프로필별 `llm_judge_result.csv`와 `.json`
- A~D `교차검증_종합비교보고서.md`
- 누적 Judge JSON 실행 로그
- 보고서 `assets/*.png`

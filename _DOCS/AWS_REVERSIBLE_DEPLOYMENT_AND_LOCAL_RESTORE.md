# AWS 회수 가능 배포와 로컬 복원 기준

> 작성일: 2026-07-18
> 적용 대상: `_Total` Docker 서비스의 향후 AWS 배포
> 상태: **설계 기준 문서 · AWS 실제 배포 및 복원은 아직 수행하지 않음**

## 1. 최우선 원칙

AWS 배포는 서비스를 올리는 것만으로 끝내지 않는다. **AWS에 실제로 올릴 때부터 나중에 모든 필요한 데이터와 설정을 로컬로 회수할 수 있게 설계한다.**

```text
로컬 Docker
→ AWS 배포
→ AWS 운영 데이터 누적
→ 백업 무결성 확인
→ 로컬 복원 시험
→ AWS 자원 종료
→ 로컬 Docker에서 과거 데이터와 함께 서비스 재개
```

AWS는 필수 운영 환경이 아니다. 필요한 기간에만 사용하고, 비용이나 운영 목적이 끝나면 로컬 Docker로 되돌아올 수 있어야 한다. AWS 자원 삭제는 로컬 복원과 과거 데이터 확인이 끝난 뒤에만 수행한다.

## 2. Docker를 AWS에 올릴 때 달라지는 범위

Docker 이미지로 서비스가 분리되어 있다면 애플리케이션 코드는 대부분 유지할 수 있다. 변경의 중심은 코드보다 운영 환경이다.

- `localhost`를 컨테이너 서비스명 또는 AWS 내부 주소로 변경
- 도메인·HTTPS·로드밸런서 구성
- Security Group과 접근 가능 포트 제한
- API 키를 환경변수 또는 AWS Secrets Manager로 관리
- 컨테이너 밖의 영구 저장소 연결
- 서버 재시작 후 컨테이너 자동 복구
- 백업·복원·모니터링·알림 정책 추가
- 관리자, QA, K6 부하 시험 기능의 접근 권한 분리

초기 소규모 운영은 `EC2 + Docker Compose`가 현재 로컬 구조를 가장 적게 바꾸는 방식이다. ECS/Fargate로 전환하면 애플리케이션 이미지는 유지할 수 있지만 작업 정의, 서비스, 네트워크, 로드밸런서와 저장소 설정을 AWS 형식으로 다시 작성해야 한다. 규모가 크지 않다면 EKS는 우선 선택하지 않는다.

## 3. 데이터는 어디에 저장되는가

Grafana는 모니터링 원본 데이터를 저장하는 서비스가 아니다. Grafana는 Prometheus 등의 데이터 원본을 조회해 그래프로 표시한다.

```text
서비스 지표
→ Prometheus가 시계열 데이터 저장
→ Grafana가 Prometheus를 조회해 그래프 표시
```

기본 Docker 구성에서는 Prometheus 데이터가 Docker 볼륨에 저장된다. AWS에 올렸다고 자동으로 S3에 저장되지는 않는다.

| 자료 | AWS 권장 위치 | 로컬 회수 형태 |
|---|---|---|
| Docker 이미지 | ECR, 원본은 Git | 같은 커밋으로 이미지 재빌드 또는 이미지 내보내기 |
| Prometheus 시계열 | EBS의 Prometheus 데이터 볼륨 | Prometheus 스냅샷 또는 정상 종료 후 데이터 디렉터리 |
| Prometheus 장기 백업 | S3 | 스냅샷 압축본과 체크섬 |
| Grafana 대시보드 | Git의 프로비저닝 JSON | 저장소에서 그대로 복원 |
| Grafana 사용자·설정 | Grafana DB 또는 외부 DB | DB 백업과 데이터 소스 설정 |
| 애플리케이션 로그·보고서 | S3 또는 EBS/EFS | 원본 폴더 구조를 유지한 다운로드본 |
| 관계형 데이터베이스 | RDS | 스냅샷과 논리 백업 파일 |
| API 키·비밀값 | Secrets Manager 등 | 평문 백업 금지, 로컬 `.env`에 별도 재설정 |

소규모 프로젝트의 기본 권장은 다음과 같다.

> Prometheus 실시간 데이터는 EBS에 저장하고, 종료 전 Prometheus 스냅샷을 만들어 S3에 백업한다. Grafana 대시보드는 Git으로 관리하고 로그·보고서는 S3에도 보존한다.

Prometheus는 기본 상태에서 S3를 주 저장소로 직접 사용하지 않는다. S3 기반 장기 시계열 저장이 필요하면 Thanos나 Grafana Mimir 같은 별도 구성이 필요하다. 현재 규모에서는 복잡도를 늘리기보다 EBS와 정기 스냅샷·S3 백업 조합을 우선한다.

## 4. 처음부터 회수 가능하게 만드는 배포 조건

AWS 배포 전에 다음 조건을 만족해야 한다.

1. 이미지 버전 고정
   - `latest`만 사용하지 않고 Git 커밋과 이미지 태그를 연결한다.
2. 영구 데이터 외부화
   - 컨테이너 내부 파일만 믿지 않고 Prometheus·Grafana·로그·보고서를 영구 저장소에 둔다.
3. 경로와 주소 환경변수화
   - AWS와 로컬에서 코드 수정 없이 저장 경로와 서비스 주소만 교체할 수 있게 한다.
4. 백업 자동화
   - Prometheus 스냅샷, Grafana 설정, 로그·보고서를 일정 주기로 백업한다.
5. 백업 목록과 체크섬 기록
   - 파일 존재 여부뿐 아니라 크기, 생성 시각, 체크섬과 대상 서비스 버전을 기록한다.
6. 로컬 복원 절차 사전 검증
   - AWS를 내리는 날 처음 복원하지 않고 운영 중 미리 복원 시험을 수행한다.
7. 삭제 승인 분리
   - 백업과 복원 작업은 진행할 수 있어도 AWS 자원 삭제는 별도 최종 승인 후 수행한다.

## 5. AWS에서 로컬로 돌아오는 표준 절차

### 5.1 자원과 데이터 목록 확정

- EC2·ECS와 실행 중인 컨테이너
- EBS·EFS·RDS 저장소
- S3 버킷과 객체 수
- Prometheus 보존 기간과 가장 오래된 시계열
- Grafana 대시보드·데이터 소스·사용자 설정
- CloudWatch 로그와 별도 보고서
- ECR 이미지 태그와 Git 커밋
- Elastic IP, ALB, NAT Gateway 등 비용 발생 자원

### 5.2 쓰기 작업 정지

복원 시점이 서로 어긋나지 않도록 신규 요청과 모니터링 수집을 잠시 중단한다. Prometheus 데이터 디렉터리를 실행 중에 무단 복사하지 않는다.

### 5.3 백업 생성

1. Prometheus 스냅샷 생성
2. Grafana 프로비저닝 파일과 DB 백업
3. 로그·보고서 동기화
4. RDS 사용 시 스냅샷과 논리 백업 생성
5. 사용한 Docker 이미지 태그와 Git 커밋 기록
6. 백업 manifest와 체크섬 생성
7. S3 또는 별도 안전한 위치에 이중 보관

### 5.4 AWS를 유지한 상태에서 로컬 복원

AWS 자원을 아직 삭제하지 않은 상태에서 백업을 로컬로 내려받아 복원한다.

1. AWS와 호환되는 Prometheus·Grafana 버전 준비
2. 로컬 Docker 볼륨 생성
3. Prometheus 스냅샷을 로컬 데이터 볼륨에 복원
4. Grafana 대시보드와 DB·데이터 소스 복원
5. 로그·보고서를 로컬 `_OUTPUT` 구조에 복원
6. Grafana 데이터 소스를 로컬 Prometheus 주소로 연결
7. AWS 운영 당시 날짜 범위로 Grafana 조회 기간 확장

### 5.5 복원 검증

- 로컬 Prometheus가 정상 시작되는가
- 가장 오래된 예상 시각과 마지막 AWS 운영 시각의 지표가 존재하는가
- Grafana의 기존 패널이 오류 없이 표시되는가
- 조회 기간을 과거로 넓혔을 때 AWS 운영 당시 그래프가 보이는가
- 주요 지표명과 라벨이 유지되는가
- 로그·보고서의 개수와 체크섬이 manifest와 일치하는가
- 로컬 서비스가 신규 데이터를 기존 데이터 뒤에 이어서 기록하는가

이 검증에 성공하면 AWS에서 수집했던 과거 Prometheus 데이터도 로컬 Grafana에서 계속 조회할 수 있다. 단, AWS 운영 중 Prometheus 보존 기간이 지나 이미 삭제된 데이터는 복원할 수 없다.

### 5.6 AWS 자원 종료

로컬 복원 검증 결과와 AWS 종료 대상 목록을 사용자에게 제시하고 최종 승인을 받는다. 승인 후 다음 순서로 종료한다.

1. 애플리케이션·모니터링 서비스 중지
2. 최종 백업과 체크섬 재확인
3. EC2·ECS·RDS 등 실행 자원 종료 또는 삭제
4. EBS·EFS 스냅샷 보존 정책 확인 후 정리
5. ALB·NAT Gateway·Elastic IP 등 독립 비용 자원 정리
6. 사용하지 않는 ECR 이미지·CloudWatch 로그 정책 확인
7. S3 백업은 정해진 보존 기간 동안 유지
8. AWS 비용 탐색기로 잔여 비용 발생 자원 재확인

## 6. Grafana 과거 그래프 복원 조건

다음 조건을 만족하면 AWS 종료 후 로컬 Grafana에서도 과거 그래프를 표시할 수 있다.

- Prometheus 시계열 데이터가 스냅샷에 포함됨
- 로컬 Prometheus 버전이 데이터와 호환됨
- Grafana 대시보드의 지표명과 라벨 쿼리가 유지됨
- Grafana 데이터 소스가 로컬 Prometheus를 가리킴
- Grafana 조회 기간을 AWS 운영 날짜까지 넓힘
- 시스템 시간대와 표시 시간대가 의도대로 설정됨

Grafana 화면만 내보내는 것으로는 과거 그래프를 복원할 수 없다. 반드시 Prometheus 원본 시계열 데이터도 함께 회수해야 한다.

## 7. AWS 관리형 모니터링을 선택할 경우

Amazon Managed Service for Prometheus나 Amazon Managed Grafana를 사용하면 운영 편의성은 높아지지만 로컬 회수 절차가 단순한 Docker 볼륨 복사와 달라진다. 관리형 서비스 선택 전 다음을 별도로 검증한다.

- 원본 시계열을 원하는 범위와 형식으로 내보낼 수 있는가
- 로컬 Prometheus에 과거 데이터를 재구성할 수 있는가
- Grafana 대시보드와 권한 설정을 내보낼 수 있는가
- 서비스 종료 전 데이터 추출에 필요한 시간과 비용은 얼마인가

로컬 복귀 가능성을 최우선으로 한다면 초기에는 직접 관리하는 Prometheus + EBS + S3 백업 구조가 더 단순하다.

## 8. Codex에 작업을 맡길 때의 진행 범위

AWS 접근 권한과 대상 범위가 제공되면 다음 작업을 순차적으로 수행할 수 있다.

- AWS 자원과 데이터 저장 위치 조사
- 백업·복원 계획과 자동화 스크립트 작성
- Prometheus·Grafana·로그·보고서 백업
- 백업 체크섬과 manifest 검증
- 로컬 Docker 복원
- 과거 Grafana 그래프 표시 확인
- AWS 종료 후보 자원과 예상 영향 정리
- 사용자 최종 승인 후 AWS 자원 종료
- 종료 후 비용 발생 자원 재확인

AWS 자원 삭제 권한은 백업 권한과 분리하는 것을 권장한다. 백업과 로컬 복원 검증 전에는 AWS 자원을 삭제하지 않는다.

## 9. 최종 완료 기준

AWS 배포는 다음 조건을 모두 만족해야 회수 가능한 배포로 본다.

- AWS에서 서비스가 정상 동작함
- 영구 데이터가 컨테이너 밖에 저장됨
- 정기 백업과 체크섬이 생성됨
- 로컬 복원 절차가 문서와 자동화로 준비됨
- 실제 로컬 복원 시험에서 과거 Prometheus 데이터가 조회됨
- Grafana에서 AWS 운영 당시 그래프가 표시됨
- AWS 삭제 전에 최종 백업과 사용자 승인이 있음
- AWS 종료 후에도 필요한 S3 백업과 로컬 데이터가 남음
- 잔여 비용 발생 자원이 없는지 재확인함

# 대시보드 K6 부하 테스트 화면 기준

> 작성일: 2026-07-17  
> 상태: **구현·비API 회귀·실제 브라우저 화면 검증 완료**  
> 적용 대상: `_Total` 통합 Streamlit 대시보드  
> 기준 기능: `tools/qa_control/main.py`의 기존 QA 시험 실행 기능

## 1. 목적

QA 컨트롤러에서 제공하는 주요 K6·성능·검증 시험 7개를 통합 대시보드에서도 실행할 수 있게 한다. 사용자는 시험 목적과 준비 상태를 먼저 확인하고, 카드에서 시험을 선택한 뒤 실행·중지하며, 고정 높이 터미널에서 진행 내용을 실시간으로 확인한다.

이 문서는 화면·동작 기준과 현재 구현 상태를 함께 기록한다. 실제 외부 AI API와 실제 고부하 시험은 실행하지 않고 실행기 단위검사, 전체 비API 회귀와 브라우저 화면으로 검증했다.

## 2. 최상위 탭 배치

왼쪽 운영 영역 4개는 유지하고 오른쪽 테스트 영역을 2개에서 3개로 확장한다.

```text
왼쪽 운영 영역
[AI 에이전트 챗봇] [VOC 챗봇] [모니터링] [보고서 모음]

오른쪽 테스트 영역
[K6 부하 테스트] [AI 에이전트 테스트케이스] [VOC 테스트케이스]
```

- `K6 부하 테스트`는 오른쪽 테스트 영역의 첫 번째 탭이다.
- `K6 부하 테스트` 안에는 별도의 하위 탭을 만들지 않는다.
- 기존 AI·VOC 테스트케이스 탭의 기능과 하위 탭은 변경하지 않는다.
- 화면 폭이 좁으면 최상위 탭 영역만 가로 스크롤하고 탭 순서를 유지한다.

## 3. 시험 카드 구성

VOC 테스트케이스의 A~D 카드와 같은 선택·강조 방식을 사용한다.

첫째 줄에는 다음 4개 카드를 둔다.

1. `기본 동작 시험 (Smoke Test)`
2. `일반 부하 시험 (Load Test)`
3. `무작위 요청 시험 (Random Test)`
4. `한계 부하 시험 (Stress Test)`

둘째 줄에는 다음 3개 카드를 둔다.

5. `순간 급증 시험 (Spike Test)`
6. `장애·기능 검증 시험 (Validation Test)`
7. `서버 연결 성능 종합 시험 (API Performance Test)`

각 카드에는 시험명, 영문명, 목적, 확인 가능한 항목, 적용값, 결과 저장 방식을 표시한다. 카드 하단에는 `시험 실행` 버튼을 둔다. 선택 카드는 배경색·테두리를 바꾸고 카드 밖 상단에 `선택됨` 또는 실행 상태를 표시한다.

반응형 카드 열은 1400px 이상에서 첫째 줄 4열·둘째 줄 3열, 761~1399px에서 2열, 760px 이하에서 1열을 기준으로 한다. 같은 행의 카드는 가장 긴 카드에 맞춰 높이를 통일하고 실행 버튼을 항상 아래에 둔다. 모바일 1열에서는 최소 높이를 해제해 내용만큼 자연스럽게 늘어나게 한다.

## 4. 시험별 입력값과 목적

| 시험 | 입력값 | 기본값 | 주요 확인 내용 |
|---|---|---|---|
| 기본 동작 시험 | 없음 | 가상 사용자 1명 | 기본 연결과 HTTP 200 응답 |
| 일반 부하 시험 | 최대 가상 인원, 실행 시간 | 20명, 60초 | 일상 부하의 지연·실패율·안정성 |
| 무작위 요청 시험 | 최대 가상 인원, 실행 시간 | 100명, 60초 | 불규칙한 요청 증감 대응 |
| 한계 부하 시험 | 최대 가상 인원, 실행 시간 | 100명, 120초 | 처리 한계와 부하 감소 후 회복 |
| 순간 급증 시험 | 최대 가상 인원, 실행 시간 | 200명, 60초 | 순간 폭주와 정상 상태 복구 |
| 장애·기능 검증 시험 | 없음 | 장애 모의 K6와 비AI 기능 검사 | 장애 대응과 기능 회귀 |
| 서버 연결 성능 종합 시험 | 없음 | 1명 → 10명 → 25명 독립 실행 | 실제 채팅 API 응답시간과 실패율 |

- 최대 가상 인원은 `1~999명`, 실행 시간은 `10~600초`만 허용한다.
- 모든 숫자 입력은 `[-] [숫자] [+]` 순서로 배치해 감소 버튼은 왼쪽, 증가 버튼은 오른쪽에 둔다.
- `+ / -` 조절은 최소·최댓값을 넘지 않으며, 범위를 벗어난 수동 입력은 가장 가까운 경곗값으로 자동 보정한다.
- `+ / -`를 짧게 누르면 1단계씩 바뀌고, 약 0.45초 이상 길게 누르면 0.1초 간격으로 자동 증감한다. 포인터를 떼거나 경곗값에 도달하면 반복을 중지한다.
- 실행 직전에 실제 적용값을 다시 표시한다.
- 기존 QA 컨트롤러와 같은 K6 스크립트·Python 실행기를 사용해 두 실행 화면의 결과가 달라지지 않게 한다.

## 5. 실행 환경 상태 표시

페이지 상단에 공통 상태 영역과 `상태 새로고침` 버튼을 둔다.

```text
실행 환경 상태
● K6 2.1.0 사용 가능
● AI 에이전트 서버 연결됨
● Prometheus 연결됨
● Grafana 연결됨
```

상태 표현은 다음 기준을 사용한다.

- 초록색: 사용 가능 또는 정상 연결
- 회색: 실행되지 않음 또는 찾을 수 없음
- 주황색: 확인 중
- 빨간색: 실행 파일 오류 또는 연결 오류

필요한 서버는 시험별로 다를 수 있으므로 각 카드에도 `실행 준비 완료` 또는 비활성화 사유를 표시한다. Grafana는 결과 조회 화면이므로 꺼져 있어도 시험 실행은 허용할 수 있지만, 실행 후 조회가 불가능하다는 안내를 표시한다. K6, 시험 대상 서버와 Prometheus가 필수인 시험은 준비되지 않았을 때 실행 버튼을 비활성화한다.

## 6. K6 설치 확인 기준

K6는 웹브라우저가 실행하지 않는다. 현재 Docker 구성에서는 **별도 K6 Runner 컨테이너**가 실행하며 Streamlit은 Runner의 Health와 K6 버전을 확인한다. 브라우저를 보고 있는 사용자 컴퓨터의 K6 설치 여부는 대시보드 실행 조건이 아니다.

Windows QA 컨트롤러의 확인 순서는 다음과 같다.

1. 프로젝트의 `RUN/k6.exe` 확인
2. 운영체제 `PATH`의 `k6` 실행 파일 확인
3. 찾은 실행 파일로 `k6 version`을 제한시간 내 실행
4. 종료 코드와 버전 문자열이 정상이면 사용 가능 처리

경로만 존재하고 실행되지 않는 파일은 설치 완료로 판단하지 않는다. K6를 찾지 못하면 실행 버튼을 잠그고 공식 설치 안내 주소를 제공한다.

### 6.1 Docker 사용 시 필수 배포 메모

> **필수:** Streamlit을 Docker 컨테이너 안에서 실행하면 Windows 호스트에 설치된 K6를 자동으로 사용할 수 없다.

- Linux 기반 Streamlit 컨테이너는 Windows의 `k6.exe`를 직접 실행할 수 없다.
- Windows 실행 파일을 단순히 컨테이너에 연결하거나 복사하는 방식으로 해결하지 않는다.
- 컨테이너가 시험을 직접 실행하는 구조라면 Streamlit 이미지 또는 같은 실행 환경에 Linux용 K6를 설치해야 한다.
- 다른 방법으로는 K6 전용 실행 서비스에 허용된 시험 ID와 설정값만 전달하는 구조를 사용할 수 있다.
- 임의 명령 문자열을 전달하는 원격 쉘 방식은 사용하지 않는다.
- Docker 이미지에 K6를 추가하면 이미지 재빌드와 컨테이너 재생성이 필요하며, 설치 버전을 문서와 화면 상태에 표시한다.

2026-07-18에 전용 실행 서비스 방식을 선택해 구현했다. Streamlit은 `K6_RUNNER_URL=http://k6-runner:8200`으로 허용된 시험 ID와 설정값만 전달하며, Runner는 Linux K6 2.1.0으로 실행한다. 로컬 Docker Desktop에서 그대로 사용하므로 AWS 배포는 필요하지 않다. 상세 구조는 `DOCKER_STREAMLIT_K6_RUNNER.md`를 따른다.

## 7. 실행 버튼 활성화 정책

실제 외부 AI API 비용 확인은 모든 시험에 공통 적용하지 않는다.

| 시험 구분 | 실행 버튼 활성 조건 |
|---|---|
| 직접 K6 5종 | K6·대상 서버·필수 기록 환경 준비 |
| 장애·기능 검증 | K6·필요 서버·검사 환경 준비 |
| 서버 연결 성능 종합 | 공통 준비 완료 + 실제 AI API 비용 확인 체크 |

`서버 연결 성능 종합 시험` 카드에만 다음 필수 확인을 표시한다.

```text
[필수 체크 사항]
□ 실제 AI API 호출과 비용 발생 가능성을 확인했습니다.
```

- 체크 전에는 서버 연결 성능 종합 시험의 실행 버튼만 비활성화한다.
- 체크 여부는 다른 6개 시험의 버튼에 영향을 주지 않는다.
- 실행이 시작되면 모든 카드의 실행 버튼을 잠가 시험 중복 실행을 방지한다.
- 체크 상태는 브라우저 세션에만 유지하고 새 세션에서 자동 체크하지 않는다.

## 8. 실행·자동 이동·터미널

시험 실행 버튼을 누르면 카드 영역 아래에 실행 화면을 확장하고 그 위치로 한 번 자동 이동한다. 이후 화면 갱신 때마다 강제로 페이지를 이동시키지 않는다.

실행 화면에는 다음 내용을 둔다.

- 현재 실행 중인 시험명
- 실행 상태와 적용 설정
- 진행 표시줄 또는 진행 단계
- `시험 중지` 버튼
- 고정 높이 실시간 터미널

터미널 높이는 360px로 고정하고 내부 스크롤로 전체 내용을 확인한다. 새 출력이 추가될 때는 자동으로 맨 아래로 이동해 최신 진행 내용을 바로 표시한다. 실행 완료·실패·중단 후에도 해당 세션의 터미널 내용을 유지한다.

Streamlit 재실행 주기 때문에 프로세스 정보가 사라지지 않도록 실행 프로세스와 출력 버퍼를 화면 렌더링 상태와 분리해 관리한다. 한 서버에서 동시에 하나의 대시보드 K6 시험만 실행하는 공통 잠금을 둔다.

## 9. 중지 처리

`시험 중지`를 누르면 다음 순서로 처리한다.

1. 버튼을 `중지 처리 중...`으로 바꾸고 중복 입력 차단
2. K6 또는 Python 부모 프로세스와 해당 실행의 자식 프로세스 종료
3. 상태를 `사용자 중단`으로 기록
4. 확보된 원문 터미널 로그 보존
5. 정상 완료용 최신 보고서·manifest 갱신 제외
6. 프로세스 종료 확인 후 다른 시험 버튼 활성화

중단·서버 종료·실행 실패 결과는 정상 완료로 표시하지 않는다. 중단된 실행이 기존 정상 정식 보고서를 덮어쓰지 않게 한다.

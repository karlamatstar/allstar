# 채팅 서비스 제어 브리지

> 작성일: 2026-07-18
> 적용 대상: Docker Streamlit의 AI·VOC 채팅 서버 재접속
> 상태: **구현·실제 Docker 수명주기·대시보드 화면 검증 완료**

## 목적

Docker 컨테이너로 전환된 Streamlit에서 기존 채팅 UX를 유지한다.

- AI 챗봇의 `채팅 서버 중단 시험`은 실제 `portfolio-api` 컨테이너를 중단한다.
- AI 채팅창 중앙의 `채팅 서버 재접속`은 `portfolio-api`를 다시 시작한다.
- 외부 원인으로 VOC 서버가 중단된 경우 중앙의 `VOC 채팅 서버 재접속`은 `voc-api`를 다시 시작한다.
- 503·504 장애 시험은 서버를 중단하지 않으므로 기존 흐름을 유지한다.

## 구조

```text
Streamlit 컨테이너
   ↓ 고정된 서비스명과 시작·중단 동작만 요청
채팅 서비스 제어 브리지
   ↓ Docker Engine API
portfolio-api 또는 voc-api
```

Streamlit에는 Docker 소켓을 연결하지 않는다. Docker 소켓은 최소 기능의 `service-control` 컨테이너 하나에만 읽기·제어 용도로 연결한다. 브리지는 Compose 내부 네트워크에서만 접근하며 호스트 포트와 Swagger 화면을 공개하지 않는다.

## 허용 범위

| 서비스 | 중단 | 시작 | 용도 |
|---|---:|---:|---|
| `portfolio-api` | 허용 | 허용 | AI 명시적 서버 중단 시험과 재접속 |
| `voc-api` | 허용 | 허용 | 외부 중단 후 VOC 재접속 |

다른 서비스명, 컨테이너 ID, 임의 Docker 명령 문자열은 받지 않는다. `service-control` 자신과 Streamlit, K6 Runner, Prometheus, Grafana, VOC 6개 에이전트는 이 API로 제어할 수 없다.

## 상태와 로그

- 브리지는 자신의 Compose 프로젝트 라벨을 Docker Engine에서 확인한다.
- 같은 프로젝트이면서 허용된 Compose 서비스 라벨이 일치하는 컨테이너만 찾는다.
- 시작·중단 응답 뒤 실제 컨테이너 상태를 다시 확인한다.
- 요청·성공·실패는 `_OUTPUT/logs/services/service_control_events.jsonl`에 누적한다.
- AI 서버 중단 시험의 질문·N/A 품질 로그·보고서와 운영 장애 카운터는 기존 흐름을 유지한다.
- VOC 재접속은 인위적 장애 시험이 아니므로 AI 장애 시험 로그를 만들지 않는다.

## 화면 동작

### AI 서버 중단 시험

1. 무작위 테스트케이스 질문을 사용자 말풍선으로 표시한다.
2. 브리지가 `portfolio-api`를 실제 중단한다.
3. 연결 실패를 `server_down`, 판정 `N/A`로 기록한다.
4. 채팅창 중앙에 서버 중단 안내와 재접속 버튼을 표시한다.
5. 채팅 입력과 장애 시험 버튼 3개를 잠근다.

### AI·VOC 재접속

1. 중앙 재접속 버튼이 브리지에 해당 API 서비스 시작을 요청한다.
2. 기존 화면 코드가 대상 `/health`의 HTTP 200을 제한시간 내 확인한다.
3. 성공하면 중앙 완료 안내를 표시하고 입력을 다시 활성화한다.
4. 실패하면 잠금 상태와 중단 안내를 유지하고 원인을 표시한다.

## 보안 경계

Docker 소켓 접근 권한은 호스트와 동등한 강한 권한이므로 다음을 지킨다.

- 서비스 제어 브리지는 외부 포트를 공개하지 않는다.
- 서비스 허용 목록을 코드 상수로 고정한다.
- 입력으로 명령어·경로·컨테이너 ID를 받지 않는다.
- Docker Engine 호출도 컨테이너 조회·시작·중단으로 제한한다.
- Streamlit과 다른 앱 컨테이너에는 Docker 소켓을 연결하지 않는다.
- AWS·외부 공개 배포에는 이 로컬 구성을 그대로 사용하지 않고 IAM·오케스트레이터 기반 최소 권한 제어로 교체한다.

## 완료 기준

- 비과금 자동검사에서 허용 목록·다른 프로젝트 차단·시작·중단·오류 처리를 확인한다.
- 실제 로컬 Docker에서 AI 서버 중단 후 8000 포트와 Health가 내려가는지 확인한다.
- 중앙 재접속 경로로 AI Health 200이 복원되는지 확인한다.
- VOC는 중단 상태에서 재접속만 검증하고 인위적 장애 로그가 생기지 않는지 확인한다.
- Streamlit·K6 Runner·Prometheus·Grafana와 VOC 에이전트가 중단 대상에 포함되지 않는지 확인한다.
- 외부 AI API를 호출하지 않는다.

## 2026-07-18 검증 결과

- Streamlit 컨테이너에서 브리지 Health `200`과 Compose 프로젝트 `total`을 확인했다.
- 브리지의 호스트 포트 `8300`은 공개되지 않으며 호스트 요청 결과가 연결 불가임을 확인했다.
- 허용 목록 밖의 `grafana` 중단 요청은 HTTP `403`으로 거부됐다.
- 브리지로 `portfolio-api`를 중단한 뒤 호스트 `8000/health`가 연결 불가가 되고, 다시 시작한 뒤 HTTP `200`으로 복원되는 것을 확인했다.
- 같은 방식으로 `voc-api`의 중단·복구와 VOC 재접속이 AI 장애 시험 로그를 추가하지 않는 것을 확인했다.
- 실제 대시보드에서 무작위 질문 표시, 중앙 중단 안내, 입력·장애 버튼 잠금, 재접속 버튼, 완료 안내 자동 전환을 확인했다.
- 전용 최소 패키지만 설치하는 `service-control` 이미지를 다시 빌드하고 내부 Health를 확인했다.
- 집중 자동검사 `17개`와 외부 AI 실행 경로를 제외한 전체 회귀 `294개`가 통과했고, `1개`는 환경 조건으로 건너뛰고 `2개`는 선택에서 제외됐다.
- 외부 AI API는 호출하지 않았다.

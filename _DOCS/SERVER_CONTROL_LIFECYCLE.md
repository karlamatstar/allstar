# 서버 관리 프로그램 종료와 서비스 정리 기준

> 구현일: 2026-07-17
> 대상: `RUN/start_servers.bat`, `tools/server_control/`

## 목적

서버 관리 프로그램을 닫았는데 Docker 서비스나 Windows 호스트의 Streamlit이 계속 남는 상황을 방지한다. 서버 관리 GUI를 실행한 사용자가 프로그램을 종료하면 이 프로젝트가 시작한 서버도 함께 종료하는 것을 기본 동작으로 한다.

## 화면 반응과 상태 확인

- Docker Desktop과 각 서비스 포트 상태는 GUI 화면 처리와 분리된 백그라운드 작업에서 확인한다.
- 포트 상태는 동시에 확인하며, 한 서비스의 연결 지연이 다른 버튼과 창 이동을 막지 않게 한다.
- 자동 상태 확인은 2.5초마다 한 번만 예약한다.
- `상태 새로고침`을 여러 번 눌러도 중복 반복 작업을 만들지 않는다.
- 이전 상태 확인이 끝나지 않았으면 다음 요청은 건너뛴다.
- 기본 창 크기는 `1440×900`, 최소 창 크기는 `1200×820`으로 유지하며 실행 시 화면 중앙에 배치한다.
- 좌측 서비스·바로가기 영역은 기본 `440px`, 최소 `420px`을 확보해 긴 한국어 버튼과 상태 표시가 밀리지 않게 한다.

## 웹 화면 바로가기

기존의 선택 서비스용 `접속` 버튼은 제거한다. `개별 시작`, `개별 종료` 아래에 다음 5개 버튼을 항상 같은 순서로 세로 배치한다.

1. AI 상담 서버 기능 명세
2. 고객 의견 분석 서버 기능 명세
3. 통합 대시보드
4. 상태 정보 수집(Prometheus)
5. 운영 상태 화면(Grafana)

각 버튼은 해당 포트가 실행 중이면 `🟢`, 중지 상태면 `⚪` 아이콘을 표시한다. 중지 상태의 버튼도 비활성화하지 않으며, 누르면 해당 서버를 먼저 시작하라는 한국어 안내 창을 표시한다.

상단 작업 버튼은 현재 왼쪽부터 `상태 새로고침`, `전체 시작`, `서버 전체 종료`, `Docker 포함 전체 종료` 순서로 표시한다.

## 2026-07-17 상단 종료 버튼 구현 결과

> 아래 요구사항은 구현과 외부 실행 대체 테스트를 완료했다.

상단 버튼을 다음 4개로 변경한다.

1. 상태 새로고침
2. 전체 시작
3. 서버 전체 종료
4. Docker 포함 전체 종료

- 기존 `전체 종료`는 `서버 전체 종료`로 이름을 바꾼다.
- `서버 전체 종료`는 Streamlit과 현재 프로젝트 Docker Compose 서비스만 종료하고 Docker Desktop은 유지한다.
- `Docker 포함 전체 종료`는 프로젝트 서비스를 먼저 정리한 뒤 Docker Desktop까지 종료한다.
- Docker 포함 종료 전에는 다른 프로젝트 컨테이너까지 영향을 받을 수 있다는 확인창을 표시한다.
- AllStar 이외의 실행 중 컨테이너가 있으면 개수를 포함한 추가 경고를 표시한다.
- Docker 포함 종료 버튼은 맨 오른쪽에 경고 색상으로 배치한다.
- 우상단 `X`와 강제 종료 감시기의 기본 동작은 `서버 전체 종료`와 같으며 Docker Desktop을 종료하지 않는다.
- 종료 중에는 중복 실행을 막기 위해 상단 작업 버튼을 비활성화하고 결과를 한국어 로그와 알림으로 표시한다.

## 전체 시작과 Docker Desktop

실행 서비스 목록 첫 항목에 `컨테이너 실행 환경(Docker Desktop)`을 표시한다. `전체 시작`은 다음 순서로 동작한다.

1. Docker 엔진 준비 상태를 확인한다.
2. 꺼져 있으면 `docker desktop start --detach`로 Docker Desktop 실행을 요청한다.
3. 최대 120초 동안 Docker 엔진이 실제 요청을 받을 수 있는 상태가 될 때까지 기다린다.
4. 준비가 끝난 뒤 `docker compose up -d --build`로 현재 코드가 반영된 이미지를 빌드하고 프로젝트 서비스를 시작한다.
5. Docker 서비스 시작이 성공하면 Windows 호스트 Streamlit을 시작한다.

Docker Desktop 시작 또는 준비가 실패하면 서버 시작을 진행하지 않고 `_OUTPUT/logs/services/shutdown_guard.log`에 원인을 기록한다. Docker Desktop 항목을 선택하면 개별 시작과 개별 종료도 가능하다.

Docker Compose 서비스의 `개별 시작`은 Docker Desktop을 자동으로 켜지 않는다. Docker가 꺼져 있으면 명령을 실행하지 않고 오른쪽 실행 기록 화면에 Docker Desktop을 먼저 시작하거나 `전체 시작`을 사용하라는 한국어 경고를 표시한다.

## 정상 종료

우상단 `X` 버튼을 누르면 GUI는 즉시 사라지지 않고 다음 순서로 처리한다.

1. 중복 종료 요청을 차단하고 창 제목을 `서버 전체 종료 중`으로 변경한다.
2. 실행 중인 Streamlit 프로세스와 그 하위 프로세스를 종료한다.
3. 현재 `_Total` 프로젝트의 `docker compose stop`을 실행한다.
4. 정상 종료 표식을 기록한다.
5. GUI 창을 닫는다.

정리 결과는 `_OUTPUT/logs/services/shutdown_guard.log`에 기록한다.

Streamlit은 GUI가 저장한 PID와 8501 포트를 실제로 수신 중인 PID를 함께 확인한다. 이전 실행에서 남은 프로세스처럼 PID 상태 파일이 없더라도 8501 포트를 점유한 Streamlit을 종료할 수 있다.

## 강제 종료 감시

`RUN/start_servers.bat`는 GUI를 직접 실행하지 않고 독립 종료 감시기 `tools/server_control/shutdown_guard.py`를 실행한다. 감시기가 GUI를 시작하고 종료 상태를 기다린다.

- GUI가 정상 종료 표식을 남기면 추가 종료 작업 없이 감시기를 끝낸다.
- 작업 관리자 등으로 GUI만 강제 종료되어 정상 표식이 없으면 감시기가 남은 Streamlit과 Docker Compose 서비스를 대신 종료한다.
- 실행 중인 Streamlit PID는 `_OUTPUT/logs/services/runtime/`의 임시 상태 파일로 전달한다.
- 감시가 끝나면 임시 상태 파일과 정상 종료 표식을 삭제한다.

## 보장 범위와 한계

- 우상단 `X`와 GUI 프로세스만 강제로 종료하는 일반적인 작업 관리자 종료는 처리할 수 있다.
- GUI와 독립 감시기를 동시에 종료하거나 전체 프로세스 트리를 강제 종료하면 정리 작업을 보장할 수 없다.
- 컴퓨터 전원 차단, 운영체제 강제 종료, Docker Desktop 자체 오류에서는 종료 작업을 보장할 수 없다.
- 감시기는 다른 프로젝트의 컨테이너를 종료하지 않고 현재 `_Total` 폴더의 Docker Compose 서비스만 정리한다.
- `서버 전체 종료`와 GUI 종료는 현재 프로젝트 서비스만 끄고 Docker Desktop 자체는 유지한다.
- `Docker 포함 전체 종료` 또는 Docker Desktop 항목의 `개별 종료`를 선택한 경우에만 Docker Desktop 자체를 종료한다.
- 8501 포트는 이 프로젝트의 Streamlit 전용 포트로 간주한다. 다른 프로그램이 8501을 사용 중이면 전체 종료 대상에 포함될 수 있다.
- 서버 관리 GUI를 `main.py`로 직접 실행하면 `X` 정상 종료는 동작하지만 강제 종료 감시는 적용되지 않는다. 강제 종료 보호가 필요하면 `RUN/start_servers.bat` 또는 `RUN/start_servers_hidden.vbs`를 사용한다.

## 검증 기준

- 런처가 GUI 대신 종료 감시기를 시작하는지 확인한다.
- GUI에 Windows 창 닫기 처리기가 등록되어 있는지 확인한다.
- 종료 함수가 기록된 Streamlit PID의 프로세스 트리와 현재 프로젝트의 Docker Compose 서비스를 정리하는지 외부 실행을 대체한 테스트로 확인한다.
- PID 기록이 없어도 8501 포트 점유 프로세스를 찾아 종료하는지 확인한다.
- Docker가 꺼진 조건에서 Docker Desktop 실행 요청 후 준비 완료를 기다리는지 외부 실행을 대체한 테스트로 확인한다.
- 실제 종료 시험을 수행할 때는 다른 작업이 해당 서버를 사용하고 있지 않은지 먼저 확인한다.
